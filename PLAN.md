## 1. Проблема и пользователь

**Запрос (что сказала Айгерим):** хаос с записью, двойные брони, мастера не в курсе, неявки, сгорающие слоты, гугл-календарь не зашёл, не хочет большую CRM.

**Настоящая боль (что под запросом):**
Айгерим теряет **деньги** (пустые слоты от неявок и поздних отмен) и **нервы** (ручная синхронизация переписки → бумажного журнала → головы трёх мастеров). Корень — нет единого источника правды о расписании. Запись живёт в 3 местах (WhatsApp, директ, журнал) и в голове, поэтому неизбежны накладки и потери.

**Пользователи и роли:**
- **Владелица/админ (Айгерим)** — главный пользователь. Хочет перестать вести запись вручную и видеть деньги/загрузку. Платит за продукт.
- **Мастер (×3)** — хочет в телефоне видеть «что у меня сегодня», без CRM-обвеса.
- **Клиентка** — хочет записаться за 30 секунд без переписки и переспрашиваний «а на 15:00 свободно?».

**JTBD (главная работа продукта):**
«Когда клиентка хочет записаться — дай ей занять реально свободный слот за 30 секунд без переписки, так чтобы Айгерим не трогала журнал, мастер видел свой день, а освободившийся слот не сгорал».

**Приоритет болей (какую берём первой и почему):**
1. **Накладки + ручной ввод** → берём первой. Это корень: убери переписку как канал записи и сделай единый календарь с защитой от пересечений — исчезает и двойная бронь, и «мастер не знал». Самый дешёвый и самый острый wedge.
2. **Неявки** → вторая. Напоминания + (позже) предоплата/депозит.
3. **Сгорающие слоты от отмен** → третья. Лист ожидания + автозаполнение.

Логика: пока запись идёт через переписку, любые напоминания и листы ожидания бессмысленны — нет источника правды. Сначала переносим запись в систему, потом наслаиваем борьбу с потерями.

## 2. Скоуп и решение

**Идея решения (как снимаем боль):**
Одна **ссылка-запись** студии (в bio инстаграма / в ответ в WhatsApp). Клиентка выбирает услугу → мастера (или «любого») → видит только реально свободные слоты → оставляет имя+телефон → готово. Запись падает в **единый календарь**, который физически не даёт двойную бронь. Мастер открывает свою страницу в телефоне и видит день. Система шлёт напоминание → меньше неявок. Отмена в один тап освобождает слот.

**v1 — что строим:**
- Публичная страница записи студии (самозапись клиентки).
- Единый календарь + защита от накладок (нельзя записать двоих в пересекающийся интервал у одного мастера).
- Рабочие часы и услуги (длительность) на мастера.
- День мастера / список записей для админа (мобайл).
- Отмена записи клиенткой по ссылке.
- Напоминание (в прототипе — заглушка/симуляция отправки).

**Что НЕ строим в v1 (смелость выкинуть):**
- Онлайн-оплату, кассу, чеки.
- Полноценную CRM (карточки клиентов, история, теги, лояльность).
- Аналитику/отчёты, зарплаты мастеров, склад материалов.
- Мультифилиалы, роли с правами, импорт.
- Нативные приложения. Только web (PWA-friendly).
- Прямую интеграцию с WhatsApp/Instagram API (v1 — просто ссылка, которую кидают вручную).

**Допущения (зафиксированы явно):**
- Одна студия, ≤ ~5 мастеров, услуги фиксированной длительности.
- Слоты с шагом 15/30 мин; буфер между записями настраивается (по умолчанию 0).
- Один часовой пояс (Asia/Almaty).
- Клиентка идентифицируется телефоном, без регистрации/пароля.
- Язык интерфейса — русский.

**Компромиссы:**
- Нет авторизации клиента → отмену защищаем уникальной ссылкой, а не аккаунтом.
- Напоминания в прототипе не уходят реально (нет смс/wa-провайдера) — показываем механику.

## 3. Требования и user flow

### Функциональные требования
- FR1. Публичная страница студии со списком услуг и мастеров.
- FR2. Расчёт доступных слотов = рабочие часы − существующие записи − перерывы, нарезанные под длительность услуги.
- FR3. Создание записи с **атомарной** проверкой на пересечение (защита от гонки/двойной брони).
- FR4. Подтверждение записи + ссылка на управление (отмена).
- FR5. Экран мастера/админа: записи на день, по мастерам.
- FR6. Отмена записи (клиенткой и админом) → слот снова свободен.
- FR7. Напоминание за N часов (заглушка в прототипе).

### Нефункциональные / ограничения
- NFR1. Никогда не показывать слот, который приведёт к накладке (источник правды — БД, проверка на сервере).
- NFR2. Мобайл-фёрст, страница записи грузится быстро, без регистрации.
- NFR3. Прозрачные состояния: занято/выбрано/подтверждено, понятные ошибки («этот слот только что заняли»).
- NFR4. Идемпотентность создания записи (двойной тап не плодит дубли).

### User flow по ролям

**Клиентка (основной флоу — он же прототип):**
1. Открывает ссылку студии.
2. Выбирает услугу (видит длительность и цену).
3. Выбирает мастера или «любого свободного».
4. Видит дни/время — только реально свободное.
5. Вводит имя + телефон.
6. Подтверждает → экран успеха + кнопки «добавить в календарь»/«отменить».

**Мастер:** открывает свою ссылку → видит сегодня/завтра списком → тапает запись → детали клиента → может отметить «пришёл/не пришёл».

**Админ (Айгерим):** видит все записи всех мастеров, может создать/отменить/перенести вручную, настроить услуги и рабочие часы.

### Краевые случаи (покрываем)
- **Гонка за слот:** двое жмут «подтвердить» на один слот → выигрывает первый, второму честная ошибка + предложение ближайших слотов.
- **Отмена клиенткой:** слот освобождается, (v2) уходит первому из листа ожидания.
- **Неявка:** админ/мастер помечает no-show (учёт для будущей борьбы: депозиты).
- **Выходной/отпуск мастера:** блокировка дат → его слоты исчезают из выдачи.
- **Перерыв/обед:** интервал недоступности внутри дня.
- **Запись в прошлое / слишком впритык:** запрещаем слоты раньше «сейчас + буфер».
- **Услуга длиннее остатка рабочего дня:** не показываем слот, который не помещается до конца смены.

## 4. Нарезка на релизы и модули

**Модули продукта:**
- A. Каталог (услуги, цены, длительности)
- B. Расписание (рабочие часы, выходные, перерывы)
- C. Движок доступности (расчёт свободных слотов)
- D. Бронирование (создание/отмена, анти-накладка)
- E. Витрина клиента (публичная страница записи)
- F. Кабинет мастера/админа (день, список, ручные действия)
- G. Уведомления (напоминания, подтверждения)
- H. Анти-потери (лист ожидания, депозиты, no-show)
- I. Каналы (WhatsApp/Instagram интеграции, аналитика)

| Релиз | Что входит | Модули | Зачем именно это |
|------|------------|--------|------------------|
| **v1 / Now** | Самозапись по ссылке, единый календарь без накладок, день мастера, отмена, напоминание-заглушка | A, B, C, D, E, F (+G заглушка) | Убирает корень боли: переписку и двойные брони. Минимум, который уже даёт ценность Айгерим. |
| **v2 / Next** | Реальные напоминания (WA/SMS), лист ожидания + автозаполнение освободившихся слотов, отметка no-show | G, H | Бьёт по деньгам: неявки и сгорающие слоты. Имеет смысл только когда запись уже в системе. |
| **v3 / Later** | Депозиты/предоплата, интеграции с WhatsApp/Instagram, аналитика загрузки, постоянные клиенты | H, I | Удержание и рост. Требует доверия и данных, накопленных в v1–v2. |

**Логика нарезки:** v1 = поставить источник правды и убрать ручной хаос (без этого всё остальное не работает). v2 = монетизация боли «теряю деньги» поверх готового календаря. v3 = масштаб и каналы.

## 5. Прототип — какой флоу и почему

**Выбираем флоу: самозапись клиентки** (услуга → мастер → слот → подтверждение).

Почему именно он:
- Это **wedge** — точка, где умирают сразу все боли (хаос переписки, двойные брони, «мастер не знал»).
- Это экран, которого касается клиент → на нём видно вкус и UX (главное в оценке прототипа).
- Он самодостаточен: можно показать кликабельно от начала до успеха.

Состояния, которые покажем: пустой выбор → выбор услуги → выбор мастера → сетка времени с занятыми/свободными → ввод контактов → успех → (ошибка «слот заняли»).

## 6. План сборки (фронт + бэк)

**Стек (рекомендация для скорости вайбкодинга, один репозиторий):**
- Next.js 15 (App Router, TypeScript) — фронт и бэк в одном.
- Tailwind + shadcn/ui — быстрый аккуратный UI.
- Prisma + SQLite локально → Postgres (Neon) на проде.
- Деплой: Vercel.

**Почему так:** один репо = фронт + бэк + БД, мгновенный деплой со ссылкой (ссылка обязательна для сдачи), не нужно поднимать отдельный сервер.

### Фазы (каждая фаза оставляет рабочий результат)

- [ ] **Ф0. Каркас.** `create-next-app`, Tailwind, shadcn, Prisma, git.
- [ ] **Ф1. Модель данных.** `Master`, `Service`, `WorkingHours`, `Appointment`, `Client`. Сид-данные: студия Айгерим, 3 мастера, 3–4 услуги, рабочие часы.
- [ ] **Ф2. Бэк — движок доступности.** Функция `getAvailableSlots(masterId|any, serviceId, date)` = рабочие часы − записи − перерывы, нарезка под длительность. API-роут/server action.
- [ ] **Ф3. Бэк — бронирование.** `createAppointment` с транзакцией + проверкой пересечения (анти-двойная-бронь), идемпотентность. `cancelAppointment`.
- [ ] **Ф4. Фронт — флоу клиента (ПРОТОТИП).** Шаги: услуга → мастер → дата/время → контакты → успех. Мобайл-фёрст, состояния загрузки/ошибки.
- [ ] **Ф5. Фронт — день мастера/админа.** Список записей на день, отмена. (минимально)
- [ ] **Ф6. Деплой.** Vercel + Neon, сид на проде, проверить ссылку с телефона.

**Если время кончается:** Ф0→Ф4 + деплой = уже сдаваемый кликабельный флоу. Ф5 — бонус, показывает «другую сторону» продукта.

### Модель данных (черновик)
```
Master      { id, name, color, isActive }
Service     { id, name, durationMin, priceKzt }
WorkingHours{ id, masterId, weekday, startMin, endMin }   // + breaks/timeoff позже
Client      { id, name, phone }
Appointment { id, masterId, serviceId, clientId, startsAt, endsAt, status }
            status: booked | cancelled | no_show | done
```
Ключевая инварианта: для одного `masterId` интервалы `[startsAt, endsAt)` со статусом `booked` не пересекаются. Проверять в транзакции при создании.
